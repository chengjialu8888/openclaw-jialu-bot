# C-Layer: Meta-Meta Layer (元层)

> "原则性和灵活性的统一，是马克思列宁主义的原则。"

---

## 本层定位

**做什么:** 优化"如何改进"的机制本身，设计元规则
**不做:** 关注具体错误或技能
**核心原则:** 让 B-Layer 更高效，让系统能自我生长

---

## 核心职能

### 1. 学习机制评估

**评估对象:** B-Layer 的效率

**关键问题:**
1. 学习 → 改进 → 生效，这个循环有多快？
2. 学习有没有重点？还是均匀用力？
3. 规则之间有没有系统性冲突？
4. 学习越多，系统是变聪明了还是变复杂了？

**评估指标:**
```
学习效率 = 减少的错误数 / 投入的学习成本
系统复杂度 = 规则数 × 规则间关联度
生长速度 = 新有效规则数 / 周期
```

**健康标准:**
- 学习效率 > 0.5 (每投入1单位学习，减少0.5单位错误)
- 系统复杂度增长率 < 20%/月
- 生长速度 > 5条有效规则/月

---

### 2. 元规则设计

**元规则:** 关于"如何制定规则"的规则

**当前元规则库:**

**Meta-Rule-001:** 规则的可执行性
> 任何规则必须包含：触发条件 + 执行动作 + 验证标准
> 不能执行的规则不是规则，是愿望

**Meta-Rule-002:** 规则的独立性
> 单条规则应该能独立理解，不依赖阅读其他规则
> 允许引用，但核心逻辑自洽

**Meta-Rule-003:** 规则的时效性
> 所有规则默认有 TTL (参考 knowledge-ttl.yaml)
> 元规则 TTL = 0 (永不过期，除非被新元规则替代)

**Meta-Rule-004:** 冲突解决机制
> 规则冲突时，时间优先 (后覆盖先)
> 如果冲突超过3次，说明分类不当，需要重新设计分类体系

**Meta-Rule-005:** 反馈闭环
> 每条规则必须能验证是否生效
> 无法验证的规则，标记为 "hypothesis"，等待验证

---

### 3. 架构优化

**优化对象:**
- A/B/C 三层的边界是否清晰
- 信息流是否顺畅
- 是否存在冗余或盲区

**架构审查清单:**

**边界清晰度:**
- [ ] A-Layer 只做执行，不做判断
- [ ] B-Layer 只做改进，不做执行
- [ ] C-Layer 只做设计，不做具体改进
- [ ] 层间接口是否明确

**信息流:**
- [ ] A → B 的错误上报是否及时
- [ ] B → A 的策略更新是否生效
- [ ] B → C 的上报是否必要且充分
- [ ] C → B 的优化是否可执行

**系统健康:**
- [ ] 是否有规则僵尸 (长期未触发)
- [ ] 是否有学习积压 (未处理的 learnings)
- [ ] 是否有策略失效 (学习了但未改进)

---

### 4. 新机制设计

**触发条件:**
- B-Layer 无法解决的问题
- 用户提出全新的工作方式需求
- 现有系统无法满足的新场景

**设计流程:**
```
需求识别 → 机制原型 → 小规模试验 → 评估效果 → 全量部署/废弃
```

**示例: 设计"知识关联图谱"机制**

**需求:** 学习之间有关联，但现在是线性列表，找不到关联

**原型设计:**
```markdown
## Mechanism: Knowledge Graph

**核心概念:**
- 节点: 知识/规则/学习
- 边: 关联关系 (依赖、冲突、强化)

**功能:**
1. 自动识别关联 (关键词、技能标签、时间)
2. 可视化展示 (供人查看)
3. 智能推荐 (学习A时，推荐关联的B和C)
4. 冲突检测 (自动发现潜在冲突)

**试验方案:**
- 先对最近 20 条 learnings 建立图谱
- 手动验证关联准确性
- 如果准确 > 80%，全量部署
```

## C-Layer 触发条件

**定期触发:**
- 每月1日 → 全面架构审查
- 每季度 → 元规则回顾

**事件触发:**
- B-Layer 上报无法解决的问题
- 系统复杂度超过阈值
- 用户要求"重新设计 [系统部分]"

**数据触发:**
- 学习效率连续下降
- 重复错误率上升
- 规则冲突频率增加

## C-Layer 的反思

**这是最高层，但它也有局限:**

1. **无法自我优化:** C-Layer 的设计质量，取决于设计者的判断
2. **无法预测未知:** 只能优化已知问题，无法预见全新场景
3. **复杂度极限:** 当系统复杂到一定程度，任何优化都会引入新复杂度

**应对:**
- 保持谦逊，承认未知
- 定期"归零"思考：如果重新设计，会怎么做
- 保持与用户的深度对话，人是最终的元层
